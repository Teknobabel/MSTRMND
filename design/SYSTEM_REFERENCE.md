# Mastermind — system reference

This document describes the **current** architecture, data flow, and content model as of the repo state. Use it to stay consistent when adding features or content.

## Tech stack

- **Vite 6** + **TypeScript** (strict), ES modules.
- **Web UI**: DOM screens for menus (`index.html` + `src/navigation.ts`); **Canvas 2D** for the placeholder game view (`src/canvas/setup.ts`, `src/main.ts`).
- **Content**: JSON files under `content/`, validated with **Zod** in `src/game/contentSchema.ts` (issue-collecting pipeline; see **Content catalog pipeline**).
- **Validation CLI**: `npm run content:validate` runs `scripts/validate-content.ts` (Node + `tsx`; reports **every** issue in one pass). **`npm run typecheck`** runs `tsc --noEmit`. **`npm run build`** runs content validation, then typecheck, then `vite build`.
- **Tests**: **Vitest** (`npm test` / `npm run test:watch`); suites live next to sources as `src/game/*.test.ts` with shared fixtures in `src/game/testFixtures.ts` (`rawFixtureSlices`, `fixtureCatalog`, `seededRng`, `sequentialIds`).

## Content catalog pipeline

1. **Authoring**: Edit JSON under `content/`. Use stable string **`id`** fields for cross-references (lowercase slugs are typical).
2. **Loading (app)**: `loadContent()` in `loadContent.ts` static-imports every JSON file into a keyed **`RawContentSlices`** record (`{ traits, minions, agents, missions, locations, maps, assets, omegaPlans, lairs, events, organizationNames, playerProfiles, wantedLevels, balance }`) and calls **`parseCatalog(raw)`**. The slice list is defined once in **`CONTENT_MANIFEST`** (slice key → `content/<key>.json`); `scripts/validate-content.ts` iterates the same manifest. **`balance`** is a **single object**, not an array (see **Game balance**).
3. **Two validation phases** inside **`parseContentCatalog`** / **`parseCatalog`**:
   - **Shape phase** (**`parseContentSlices`**): each slice parsed **independently** against **`contentSliceSchemas[key]`** and normalized (e.g. `startingLevel` defaults to 1). A slice that fails shape is `null` in the result; its Zod errors become **`ContentIssue`** rows (`{ slice, entityId, path, message }`).
   - **Semantic phase** (**`collectContentIssues`**): array-level rules (duplicate ids, wanted-level ordering) plus the cross-slice reference checks — same logical dependencies as before: **traits** (defines `traitIds`) → **minions** (trait refs) → **agents** (same shape + trait refs; ids **disjoint** from minion template ids) → **assets** (defines `assetIds`) → **missions** (trait + asset refs + mission-effect placement rules) → **locations** → **`startingDynamicTraits` cross-check** on minion + agent templates → **maps** (location refs) → **omega plans** (mission refs + **`mapId`**) → **lairs** (mission refs + optional `startingAssets` asset refs; **`upgradeMissionIds`** disjoint from **`availableMissionIds`**) → **events** (ids **disjoint** from mission template ids; optional **`expireEffects`**; **both** requirement lists may be empty) → **wanted levels** (ordered infamy tiers). Checks whose inputs failed shape parsing are skipped; **all issues are collected**, never throw-on-first.
   - **`parseContentCatalog(raw)`** returns `{ catalog: ContentCatalog | null, issues: ContentIssue[] }` (`catalog` is null whenever issues exist). **`parseCatalog(raw)`** — used at boot and by the build gate — throws **`ContentValidationError`** whose message lists **every** issue. Content tooling should prefer the non-throwing form, the exported per-entity schemas (validate a single entity as it is edited), and **`collectContentIssues`** on a full draft catalog.

**Cross-reference rules** (enforced in `contentSchema.ts`):

- Minion `startingTraitIds` / `levelUpTraitOrder` → existing trait ids.
- Optional minion / agent **`startingDynamicTraits`**: each entry’s **`targetMinionTemplateId`** must reference an existing **minion** template id (not self); **`locationId`** must reference an existing location; no duplicate **`kind`+target** key; at most one positive bond (friend/lover) and one negative (rival/hatred) toward the same minion template; no **hero** and **wanted** for the same location on the same template.
- Agent templates (`content/agents.json`): same trait-reference rules as minions; **`id`** must not collide with any **minion** template **`id`**.
- Mission **`requiredTraitIds`** → traits; **unique** per mission. Mission **`requiredAssetIds`** → assets; **duplicates allowed** (each occurrence needs one inventory unit). **`gain_assets`** **`assetIds`** and **`exchange_assets`** **`removeAssetIds`** / **`gainAssetIds`** → assets; **duplicates allowed** (multiset). **`exchange_assets`** must have at least one id across the two lists. **At least one** of `requiredTraitIds` or `requiredAssetIds` must be non-empty per mission. Missions with **`reveal_target_asset`** or **`steal_target_asset`** in **`onSuccessEffects`** / **`onFailureEffects`** must use **`targetType`** `asset_hidden` or `asset_revealed`. **`reveal_all_hidden_assets_at_location`**, **`steal_all_assets_at_location`**, **`steal_all_revealed_assets_at_location`**, and **`security_level_delta`** require **`targetType`** `location`, `asset_hidden`, or `asset_revealed`. **`add_target_minion_traits`** requires **`targetType`** `minion`; **`traitIds`** must reference existing traits and be **unique** within the effect (≥ 1 entry). **`add_random_participant_traits`** / **`add_all_participant_traits`** **`traitIds`** → existing traits, **unique** within the effect (≥ 1 entry). **`unlock_lair_mission`** may appear only in **`onSuccessEffects`** (not **`onFailureEffects`**); **`missionId`** must reference an existing mission template.
- Omega plan **`mapId`** → **`MapTemplate.id`**; omega plans parsed **after** maps.
- Omega plan stage `missionIds` (3×3 grid) → missions; **same mission id may repeat** within a plan.
- **Lair** **`availableMissionIds`** / **`upgradeMissionIds`** → missions; no duplicate mission id **within** each list; the two lists are **disjoint** per lair; **`startingAssets`** keys → asset ids.
- **Events** (`content/events.json`): **`EventTemplate`** extends **`MissionTemplate`** with optional **`expireEffects`** (same **`MissionEffect`** union as success/failure lists; **`unlock_lair_mission`** forbidden in **`expireEffects`**). **`id`** must not collide with any **mission** template **`id`**. **`requiredTraitIds`** / **`requiredAssetIds`** are validated like missions, but **both may be empty** (unlike missions, which require at least one non-empty requirement list). Effect placement rules match missions for target-dependent kinds.
- Map `locationIds` → locations; no duplicate location id **within the same map**; a location may appear on multiple maps.
- Asset ids unique in `assets.json`.
- **`organizationNames.json`**: JSON array of at least one non-empty string (no cross-references).
- **`wantedLevels`** (`content/wantedLevels.json`): ordered array of tiers (**`minInfamy`** 0–100 strictly ascending, first tier **`minInfamy === 0`**), **`name`** (display), **`maxAgents`** (non-decreasing integers ≥ 0 — cap on opposing agents in play for that tier). Ordering rules enforced in **`collectContentIssues`**.

## Domain objects (catalog templates)

Enums in JSON are generally **lowercase** strings (e.g. trait `type`, location `locationType`).

### Traits (`content/traits.json`)

- **Trait**: `id`, `name`, `type`: `status_positive` | `status_negative` | `primary` | `secondary`.
- Passive definitions; minions, agents, and missions reference trait ids. **`status_positive`** / **`status_negative`** are excluded from per-location site/security rolls (see **Locations**). On missions, each participating minion’s trait list contributes flat success modifiers: **+10%** per **`status_positive`** occurrence and **−20%** per **`status_negative`** occurrence (summed across all participants), applied after the linear base % and clamped with the final result to **0–100**.
- **Dynamic traits** (runtime only, not catalog `Trait` rows; stored on **`MinionInstance.dynamicTraits`**; see **`dynamicTrait.ts`**): kinds **`friend`** / **`lover`** / **`rival`** / **`hatred`** (toward another roster minion **`instanceId`**) and **`hero`** / **`wanted`** (toward a **`locationId`**). Flat success modifiers (summed across participants for the mission roll and UI preview): Friend **+5%**, Lover **+10%**, Rival **−5%**, Hatred **−10%**, Hero **+5%**, Wanted **−5%**, each applying when the linked minion is on the mission or the mission’s **`getMissionTargetLocationId`** matches the location trait. After **`applyMissionEffects`** on resolve, each participant independently rolls **10%** for Friend (success, **>1** participants) or Rival (failure, **>1** participants), and for Hero (success) / Wanted (failure) when the mission has a location id; Friend→Lover and Rival→Hatred upgrade when the roll targets an existing base bond; opposite-side bonds **replace** the prior bond for that target; Hero/Wanted replace each other per location. Designer optional **`startingDynamicTraits`** on **`MinionTemplate`** (see **Minions**) is validated in **`contentSchema.ts`** after locations load.

### Minions (`content/minions.json`)

- **MinionTemplate**: `id`, `name`, `description`, **`hireCommandPoints`** (integer ≥ 0; CP spent when hiring in the Main Phase), optional `startingTraitIds`, optional **`startingLevel`** (integer ≥ 1, default **1**; grants sequential traits from **`levelUpTraitOrder`** via the same rules as automatic level-ups until **`currentLevel`** reaches this value), `levelUpTraitOrder` (ordered trait ids for further level-ups), optional **`startingDynamicTraits`**: designer-authored runtime bonds (`friend` \| `lover` \| `rival` \| `hatred` + **`targetMinionTemplateId`**, or `hero` \| `wanted` + **`locationId`**; validated for refs, self-target, duplicates, and hero/wanted or positive/negative conflicts per **`contentSchema.ts`**).
- **MinionInstance** (runtime, not in JSON): **`instanceId`** (stable id for roster and missions), `templateId`, `currentLevel`, `currentExperience`, `traitIds`, **`dynamicTraits`** (array of **`DynamicTrait`** — relationship/location modifiers separate from catalog **`traitIds`**).
- **Creation** (`minion.ts`): `createMinionFromTemplate(template, instanceId, overrides?)` — callers supply `instanceId` (e.g. `crypto.randomUUID()`).
- **Mission XP & leveling** (`minion.ts` + **`executePlan`** in `gameState.ts`): when a mission **resolves** (duration reaches 0 with valid template and participants), **each participant** gains **`MINION_XP_PER_MISSION`** (**1**) XP. When **`currentExperience`** reaches **`MINION_XP_TO_LEVEL`** (**3**), the minion **levels up** (`currentLevel` +1), **`currentExperience` resets to 0**, and **`applyLevelUp`** runs: grant the **first** trait in `levelUpTraitOrder` the instance does **not** already have (if none left, level still increases but **no** new trait). A **`minion_leveled_up`** activity event is recorded for that turn (see **Activity log** below).
- **Manual / catalog level-up helpers** (`minion.ts`): `nextLevelUpTraitId`, `applyLevelUp`, `addTrait`, `removeTrait` — same ordered trait rule as automatic level-ups.

### Agents (`content/agents.json`)

- **AgentTemplate**: Same fields as **MinionTemplate** (`types.ts` aliases `AgentTemplate` to `MinionTemplate`). **`hireCommandPoints`** is validated like minions (often **`0`** for non-hirable opposition). Trait rules match minions.
- **`AgentInstance`**: **`MinionInstance`** plus **`catalogVisibility`**: `hidden` \| `revealed` (default **`hidden`** for spawns). Not stored on **`PlayerState`**. Instances live on **`GameState.opposingAgentInstances`**; which location each belongs to is tracked in **`GameState.locationAgentPresence`** (see **Locations**). Location cards list **all** agents at the site for debug: **revealed** names plain, **hidden** names in parentheses.
- **Creation** (`agent.ts`): **`createAgentFromTemplate`** delegates to **`createMinionFromTemplate`** so leveling helpers in **`minion.ts`** apply unchanged when you spawn runtime agents later.
- **Lookup / spawn** (`agent.ts`): **`getAgentTemplateById`**, **`getOpposingAgentByInstanceId`**, **`getOpposingAgentsAtLocation`**, **`getRevealedOpposingAgentsAtLocation`** (subset helper), **`countOpposingAgentsAtLocation`** / **`countOpposingAgentsAtLocationFromData`** (success % helpers), **`revealAllOpposingAgentsAtLocation`** (runtime reveal), **`spawnOpposingAgentsAfterWantedEscalation`** (wanted tier index must **increase** that resolve; fills toward **`maxOpposingAgentsForWantedIndex`** with random unused **`catalog.agents`** template and random playable **`locationId`** from the active omega map; new instances default to **`catalogVisibility: "hidden"`**).

### Missions (`content/missions.json`)

- **MissionTemplate**: `id`, `name`, `description`, **`targetType`**: `location` | `asset_hidden` | `asset_revealed` | `minion` | `none`, **`startCommandPoints`** (≥ 0), **`requiredTraitIds`** (unique trait ids; may be empty if assets carry the requirement), **`requiredAssetIds`** (catalog asset ids; **multiset** — duplicate ids mean multiple slots; each filled slot commits one matching unit from **`PlayerState.assets`** at **`assignMission`** and is stored on **`ActiveMission.plannedAssetIds`**), **`durationTurns`** (≥ 1), optional **`onSuccessEffects`** / **`onFailureEffects`**: arrays of **`MissionEffect`** (applied in order when a mission resolves; see **`missionEffects.ts`**). **`MissionEffect`** `kind` values: `reveal_target_asset`, `reveal_all_hidden_assets_at_location`, `steal_target_asset`, `steal_all_assets_at_location`, `steal_all_revealed_assets_at_location`, `unlock_lair_mission` (`missionId`), **`gain_assets`** (`assetIds`: multiset of catalog asset ids — duplicates grant multiple units), **`exchange_assets`** (`removeAssetIds`, `gainAssetIds`: multisets — removes up to inventory holdings per id, then grants gains), **`security_level_delta`** (`delta`: signed change to **`securityLevel`** at the mission location — negative reduces — clamped to **`[0, locationLevel]`**), **`security_level_delta_global`** (signed delta applied to **every playable location**, each clamped to **`[0, that location’s locationLevel]`**), **`security_level_delta_by_location_type`** / **`security_level_delta_by_location_level`** (same clamp semantics, filtered by **`locationType`** or **`locationLevel`**), **`reveal_hidden_assets_global`** / **`reveal_hidden_assets_by_location_type`** / **`reveal_hidden_assets_by_location_level`** (reveal up to **`count`** random **hidden occupied** asset slots in scope), **`remove_trait_from_all_minions`** (**`traitId`**), **`add_trait_to_random_minions`** (**`traitId`**, **`count`** — pool = full hired roster **including busy**), **`grant_command_points_next_turn`** (**`amount`** → **`pendingBonusCommandPoints`**), **`add_success_chance_modifier`** (**`delta`**, **`turns`** → **`activeSuccessModifiers`**), **`add_target_minion_traits`** (`traitIds`: unique catalog trait ids; grants any not-already-held to the **target minion** in **`player.minions`**; mission **`targetType`** must be `minion`), **`add_random_participant_traits`** (`traitIds`: unique catalog trait ids; grants any not-already-held to one participant chosen uniformly from **`participantInstanceIds`** using the resolve-phase RNG), **`add_all_participant_traits`** (same **`traitIds`** rules; grants any not-already-held to **every** participant in **`participantInstanceIds`**), `infamy_delta` (`amount`), `max_concurrent_missions_delta`, `max_roster_size_delta`, `max_hire_offers_delta`, `max_participants_per_mission_delta`, `max_command_points_per_turn_delta` (each with signed integer **`delta`** except infamy). Baseline infamy (−3 / +5) applies first, then effect deltas; **`infamy_delta`** effects **add**; final infamy is clamped 0–100. **`orderedMissionEffects`** runs all **reveal** kinds first, then steal kinds (`steal_target_asset`, `steal_all_assets_at_location`, `steal_all_revealed_assets_at_location`), then the rest (including **`unlock_lair_mission`**, **`gain_assets`**, **`exchange_assets`**, **`security_level_delta`**, **`security_level_delta_global`**, scoped security deltas, hidden-asset reveals, **`add_target_minion_traits`**, **`add_random_participant_traits`**, **`add_all_participant_traits`**, roster trait sweeps, **`grant_command_points_next_turn`**, **`add_success_chance_modifier`**); **`steal_all_assets_at_location`** reveals hidden assets at the mission location then empties every occupied slot into inventory; **`steal_all_revealed_assets_at_location`** empties only **revealed** occupied slots; **`gain_assets`** adds to **`player.assets`** without changing locations; **`exchange_assets`** applies **`asset_lost`** / **`asset_gained`** for inventory changes; other steals leave **`empty`** slot(s). **`add_target_minion_traits`** appends each missing trait id to that minion's **`traitIds`** (no duplicates; ordering preserved); **`add_random_participant_traits`** does the same for one random roster minion in **`participantInstanceIds`**; **`add_all_participant_traits`** does the same for **all** of them; **`executePlan`** then syncs its participant lookup with updated minions so XP / level-up runs on top of new traits. **`unlock_lair_mission`** appends **`missionId`** to **`GameState.lairMissionIds`** if not already present (handled in **`executePlan`** after **`applyMissionEffects`**).
- **Success chance** (`mission.ts` + callers): **`successChancePercent(template, participants, options?)`** where **`MissionSuccessOptions`** may include **`additionalRequiredTraitIds`** (from **`missionSuccessOptionsForTarget`** in `gameState.ts`: site **required** traits + **revealed** security traits for the mission’s target location), **`playerAssets`** (inventory snapshot for the asset portion **only when** **`assignedAssetIds`** is omitted or its length does not match **`template.requiredAssetIds`**), **`assignedAssetIds`** (parallel to **`requiredAssetIds`**: each index is **`null`** or the matching catalog id; when provided with matching length, counts toward the base % instead of **`playerAssets`**), **`traitsCatalog`** (the parsed **`catalog.traits`** slice — when present, status trait modifiers apply), **`dynamicTraitDelta`** (flat sum from **`dynamicTraitSuccessModifierFromFullRoster`** in **`dynamicTrait.ts`**, using materialized pending template bonds against the full roster), **`eventSuccessModifierDelta`** (flat sum of **`delta`** from **`GameState.activeSuccessModifiers`** — timed modifiers from **`add_success_chance_modifier`**; each entry’s **`turnsRemaining`** decrements once per **`executePlan`**), and **`opposingAgentPenaltyCount`** (non-negative integer; each unit applies **−20%** after status, dynamic, and event modifiers). **Base %**: numerator = matched **distinct** required traits (participant union) + matched **asset slots** (either **`countMatchedAssignedSlots`** when **`assignedAssetIds`** matches length, else per-id `min(requiredCount, quantity)` from **`matchedAssetUnits`**); denominator = trait count + **asset occurrence count**; `Math.round(100 * matched / total)`; empty denominator → base **100%**. **Final %**: base plus **+10** per participating **`status_positive`** trait id on roster minions minus **20** per **`status_negative`** occurrence (each minion’s `traitIds` entry counts once per id), plus **`dynamicTraitDelta`**, plus **`eventSuccessModifierDelta`**, minus **20** × **`opposingAgentPenaltyCount`**, then clamped to **0–100**. **Opposing agents** (see **Locations**): only when the mission target maps to a **site** via **`getMissionTargetLocationId`** (location- or asset-targeted missions). **`executePlan`** passes the **full** agent count at that site (hidden + revealed) so the roll and **`mission_completed.successChancePercent`** reflect reality. **Plan / active mission UI** (`main.ts`) passes only **`revealed`** agents in that count so **hidden** opposition does not lower the **displayed** success %; the **Activity** line `roll … vs N%` shows the **true** resolve-time chance (including hidden agents and **dynamic** / **event** modifiers). **`minion`** / **`none`** targets omit the opposing-agent modifier (no site location). Participants: 1–**`maxParticipantsPerMission`** minions for **lair** / **omega**; **event** missions use **1–3** fixed (see **Events**).

### Events (`content/events.json`)

- **`EventTemplate`**: Same fields as **`MissionTemplate`**, plus optional **`expireEffects`**: **`MissionEffect[]` applied automatically at the end of `executePlan`** if the player **did not** assign the **`GameState.currentEventTemplateId`** offer during that turn’s Main Phase (**`eventOfferEngagedThisTurn`** stayed false). Expire uses a stub **`ActiveMission`** (no participants / no target) so participant-only random effects are no-ops. Ignoring an offer (no drag to Plan) is implicit — there is no Decline button.
- **Runtime offer**: **`GameState.currentEventTemplateId`** is one catalog **`EventTemplate.id`** (or **`null`** when **`catalog.events`** is empty), chosen at run start and **re-rolled uniformly (with replacement) at the end of every `executePlan`** after optional expire + modifier decay. **`event_rotated_in`** records the new id in the activity log.
- **Starting an event**: Assign with **`missionSource === "event"`**, **`missionTemplateId === currentEventTemplateId`**, **1–3** participants (ignores **`player.maxParticipantsPerMission`**), omega indices **`null`**. Counts toward **`player.maxConcurrentMissions`**. Same **`ActiveMission`** pipeline as lair/omega for resolve, success %, and **`applyMissionEffects`**.
- **Timed success modifiers**: Effects **`add_success_chance_modifier`** push **`{ delta, turnsRemaining }`** onto **`GameState.activeSuccessModifiers`**. Each **`executePlan`** adds the total **`delta`** into resolve rolls and UI previews via **`eventSuccessModifierDelta`**, then decrements **`turnsRemaining`** and drops expired entries.
- **Bonus CP next turn**: **`grant_command_points_next_turn`** adds to **`player.pendingBonusCommandPoints`**; **`advanceToNextTurn`** sets **`commandPoints = maxCommandPoints + pendingBonusCommandPoints`**, then clears the bonus field.
- **Global / scoped effects**: **`security_level_delta_global`**, **`security_level_delta_by_location_type`**, **`security_level_delta_by_location_level`**, and the **`reveal_hidden_assets_*`** variants iterate **playable** locations from the active omega map; security deltas clamp per site to **`[0, locationLevel]`**. **`remove_trait_from_all_minions`** strips a trait id from every hired roster minion. **`add_trait_to_random_minions`** picks up to **`count`** distinct hired minions uniformly at random (**including minions busy on missions**), grants the trait if missing.

### Mission targets (runtime + planning UI)

- **`MissionTarget`** (runtime union in `types.ts`, stored on **`ActiveMission.target`**): `location` (`locationId`), `asset` (`locationId`, `slotIndex`, `visibilityAtAssign`), `minion` (`instanceId`), or `none`. Asset targets reference a **`LocationAssetSlot`** row at run time (by index into **`GameState.locationAssetSlots`** for that location).
- **`assignMission`** validates that the chosen **`MissionTarget`** matches the mission template’s **`targetType`** (see **`missionTargetMatchesTemplate`**). **Location** and **asset** targets require a **playable** map location; **asset** also requires the slot to exist, to be **`occupied`** (not **`empty`**), and **`visibilityAtAssign`** to match the slot’s current visibility. **Minion** targets must be a roster minion not on another mission and **not** among the participants. **`none`** skips target validation.
- **Security bump** on resolve: **`+1`** at **`getMissionTargetLocationId(am.target)`** when non-null, capped at **`maxSecurityLevelForLocation`** = that site’s authored **`locationLevel`** (1–3), not a global 3 for all sites. **Location**- and **asset**-targeted missions affect that asset’s **location**; **minion** / **none** do not change security by target (`raiseSecurityAfterMissionAtLocation` in **`gameState.ts`**). **`setLocationSecurityLevel(state, catalog, locationId, level)`** clamps security for future systems that lower heat; revealed security traits follow **`securityLevel`** (see **Runtime game state**).
- **Plan mission UI** (`main.ts`): **Mission** slot and **Target** slot can be filled in any order. The target **label** updates with the mission’s **`targetType`** (e.g. “Target Location”). If **`targetType === "none"`**, the whole target field (**`#assign-target-field`**) is **hidden**. Dropping a mission that **mismatches** the staged target **clears** the target slot (**`reconcileTargetWithMission`**). **Asset** rows on location cards are **draggable chips** (per-slot identity). Roster minion drags use JSON payloads. Putting a minion in **participants** clears the same minion from the **target** slot, and vice versa. When the planned mission has **`requiredAssetIds`**, a **Required Assets** fieldset lists one drop slot per occurrence; **Assets** panel cards drag with **`mastermind-asset-card`** JSON (type-matched slot only; **Available** counts subtract units already staged in the plan).

### Locations (`content/locations.json`)

- **LocationTemplate**: `id`, `name`, `description`, `locationType` (`political` | `military` | `economic`), **`locationLevel`** (1 | 2 | 3). Per-location **asset placement** is **not** in this JSON; at run start **`createInitialGameState`** assigns **1–3** random catalog assets per **playable** location on the active map (`LocationAssetSlot`: **`kind`** `occupied` with **`assetId`** + **`visibility`** `hidden` | `revealed`, or **`kind`** `empty` after a steal), then reveals **`min(3, total slots)`** slots globally at random.
- **LocationSecurityState** (runtime): `locationId`, **`securityLevel`** (`0` | `1` | `2` | `3`). New runs start at **`0`** via **`initialLocationSecurityStatesForLocations`**. Effective maximum per site is the location’s **`locationLevel`** (enforced when raising security). **During play**, each resolved mission at that site (via **`getMissionTargetLocationId`**) increments security by **1** up to that cap.
- **Per-run site traits** (runtime, `GameState`): **`locationRequiredTraits`**: rolled at run start from **primary + secondary** traits only (excludes **`status_positive`** / **`status_negative`**) — level **1** → 0 traits, level **2** → 1, level **3** → 2. **`locationSecurityTraits`**: rolled stack of length **`locationLevel`** (reveal order = array order); only the first **`securityLevel`** entries merge into mission requirements (hidden entries do not). Both merge into **`missionSuccessOptionsForTarget`** for **location**/**asset** targets (see **`revealedSecurityTraitIds`**).
- **Opposing agents at sites** (runtime, `GameState`): **`locationAgentPresence`** lists **`AgentInstance.instanceId`** values per playable **`locationId`** (starts empty). Full **`AgentInstance`** rows live in **`opposingAgentInstances`** (each has **`catalogVisibility`**; spawns default **hidden**). Not authored in **`locations.json`**; **`spawnOpposingAgentsAfterWantedEscalation`** appends new ids to both when wanted escalates. **Locations** cards show an **Agents** row when any opposing agent is present (comma-separated template names; **hidden** names wrapped in **parentheses** for debug). Each opposing agent at the mission’s **target site** (same location id as **`getMissionTargetLocationId`**) reduces success chance by **20%** at resolve; when a mission with such a target **resolves**, every opposing agent at that site becomes **`revealed`** (success or failure).

### Maps (`content/maps.json`)

- **MapTemplate**: `id`, `name`, `description`, `locationIds` (ordered list of location ids).

### Assets (`content/assets.json`)

- **Asset**: `id`, `name`, optional `description`. Referenced by **missions** (`requiredAssetIds`), **lairs** (`startingAssets`), and **player inventory** **`PlayerState.assets`** (`Record<assetId, quantity>`). **Mission success %** uses **`assignedAssetIds`** on active missions (reserved inventory) or **`playerAssets`** when no per-slot list applies; the **Assets** panel shows **Available** (owned minus units staged in the plan mission UI).

### Lairs (`content/lairs.json`)

- **LairTemplate**: `id`, `name`, optional `description`, **`availableMissionIds`** (missions shown under the Lair **Missions** tab and assignable with **`missionSource === "lair"`**; no duplicate ids per lair), **`upgradeMissionIds`** (optional in JSON, default **[]**; **disjoint** from **`availableMissionIds`** — one-time **upgrade** missions listed for the Lair **Upgrades** tab), optional **`startingAssets`** (`Record<assetId, quantity>` merged into **`PlayerState.assets`** at run start). **`GameState.activeLairId`** is chosen at random when **`catalog.lairs`** is non-empty; **`lairMissionIds`** starts as a copy of **`availableMissionIds`** and grows when **`unlock_lair_mission`** succeeds or via **`addLairMissionToPool`**. **`completedLairUpgradeMissionIds`** tracks upgrade missions that have **resolved successfully** once this run (they disappear from the Upgrades tab).

### Omega plans (`content/omegaPlans.json`)

- **OmegaPlanTemplate**: `id`, `name`, `description`, optional **`cardArt`** (rendered in the Omega Plan panel header **only when authored** — no default placeholder), **`mapId`** (must match **`MapTemplate.id`**), `stages` — exactly **3** stages, each with **`missionIds` of length 3** (fixed 3×3 grid). Win-path content only; **grid layout** is in JSON. **Runtime progress** lives in **`GameState`**: **`activeOmegaStageIndex`**, **`omegaRowProgress`** (three booleans for the current row), updated when **`executePlan`** marks omega slots successful; advancing the stage resets row flags (see **`gameState.ts`**).

### Organization names (`content/organizationNames.json`)

- JSON **array of strings** (each non-empty). At run start **`createInitialGameState`** picks one at random for **`GameState.organizationName`** (shown at the top of the controls panel). No ids or cross-references.

### Wanted levels (`content/wantedLevels.json`)

- **`WantedLevelTier`**: **`minInfamy`** (inclusive floor, 0–100), **`name`** (shown as **Wanted** in the HUD), **`maxAgents`** (cap on opposing agents placed on the map for that tier; **`executePlan`** may spawn up to this count when the tier index **increases** after resolve).
- Runtime **`GameState.wantedLevelTierIndex`** indexes **`catalog.wantedLevels`**; updated **once per **`executePlan`**** after all resolving missions have applied infamy. **Never decreases** even if **`player.infamy`** drops. When the computed index is **greater** than the value at resolve start, **`spawnOpposingAgentsAfterWantedEscalation`** runs (see **`agent.ts`**).

### Game balance (`content/balance.json`)

**`BalanceConfig`** (`types.ts`) — a **single object** slice of designer-tunable knobs. Every field has a Zod **default** equal to the legacy constant (**`DEFAULT_BALANCE`** in `types.ts`, the single source for schema defaults and code fallbacks), so `{}` is a valid file that changes nothing. **All numeric gameplay constants quoted elsewhere in this document are these defaults** — the balance file overrides them. Parsed as `catalog.balance`; edited in the editor's **Game Balance** view (each field has a plain-language tooltip + hint).

| Knob (default) | Applied where |
|---|---|
| `statusPositiveBonus` (10) / `statusNegativePenalty` (20) / `opposingAgentPenalty` (20) | `computeSuccessChanceBreakdown` via **`MissionSuccessOptions.balance`** (resolve + UI previews) |
| `dynamicTraitModifiers` (friend +5 / lover +10 / rival −5 / hatred −10 / hero +5 / wanted −5) | `dynamicTrait.ts` modifier functions (optional `modifiers` param; breakdown-from-roster reads `catalog.balance` itself) |
| `dynamicTraitRollPercent` (10) | `rollDynamicTraitsAfterMission` (passed by `executePlan`) |
| `infamySuccessDelta` (−3) / `infamyFailureDelta` (+5) | baseline infamy in `executePlan` |
| `injuryChancePerAgentPercent` (20) | critical-failure injury rolls in `executePlan` (`min(100, per-agent × count)`) |
| `startingMaxCommandPoints` (5), `startingMaxRosterSize` (5), `startingMaxHireOffers` (3), `startingMaxConcurrentMissions` (2), `startingMaxParticipantsPerMission` (3) | `PlayerState` init in `createInitialGameState` (mission effects still mutate the live values) |
| `rerollHireOffersCp` (1) | `rerollHireOffers` + Main-Phase button |
| `eventMaxParticipants` (3) | event-mission crew cap in `assignMission` / `executePlan` / plan UI |
| `fireRehireCooldownTurns` (3) | `fireMinion` (now takes `catalog`) |
| `minionXpPerMission` (1) / `minionXpToLevel` (3) | `awardMissionResolutionExperience` (optional `xp` param, passed by `executePlan`) |
| `assetsPerLocationMin`/`Max` (1/3), `initialRevealedAssetSlots` (3) | `initializeLocationAssetPlacements` at run start |
| `securityGainPerResolvedMission` (1) | `raiseSecurityAfterMissionAtLocation` |

Semantic check: `assetsPerLocationMin ≤ assetsPerLocationMax` (in **`collectContentIssues`**). The old exported constants (`OPPOSING_AGENT_SUCCESS_PENALTY`, `MINION_XP_*`, `REROLL_HIRE_OFFERS_CP`, `EVENT_MAX_PARTICIPANTS_PER_MISSION`, `MINION_FIRE_REHIRE_COOLDOWN_TURNS`, `DYNAMIC_TRAIT_ROLL_PERCENT`) remain as **deprecated aliases of the defaults** — new code must read `catalog.balance`.

## Runtime game state (`src/game/gameState.ts`)

Not persisted in JSON; created per session via **`createInitialGameState(catalog, rng?)`** (requires **`ContentCatalog`** to roll initial hire offers; pass a seeded **`Rng`** for deterministic runs — defaults to `Math.random`). **`executePlan(state, catalog, rng, newInstanceId?)`** likewise accepts an id generator (default `crypto.randomUUID`) so resolves are fully deterministic under a seeded RNG (tests, future replays). **Resolution is simultaneous by design**: every mission resolving in one `executePlan` reads site required/security traits from the **start-of-turn snapshot**, so an earlier mission's security bump that resolve never changes a later mission's requirements (pinned by `gameState.test.ts`).

- **`TurnPhase`**: `main` | `resolve` | `summary` — **`executePlan`** sets **`summary`** briefly; the game UI immediately calls **`advanceToNextTurn`**, so the player usually sees **`main`** again after **Execute Plan** with **`commandPoints`** refilled and **`turnNumber`** incremented (no separate **Next Turn** control).
- **`PlayerState`**: `commandPoints` / `maxCommandPoints` (new game: 5/5; **`advanceToNextTurn`** sets `commandPoints = maxCommandPoints + pendingBonusCommandPoints`, then clears **`pendingBonusCommandPoints`** to **0**), `infamy` (0–100, clamped), `minions` (`MinionInstance[]`), `assets` (`Record<assetId, quantity>`), **`maxRosterSize`** (default **5**; hire blocked at cap), **`maxHireOffers`** (default **3**; cap on random offers per resolve), **`maxConcurrentMissions`** (default **2**; assign blocked when active mission count is at cap; may exceed current active count after a decrease until missions finish), **`maxParticipantsPerMission`** (default **3**; assign UI and **`canAssignParticipants`** cap for **non-event** missions), **`pendingBonusCommandPoints`** (default **0**; one-shot CP added on top of **`maxCommandPoints`** at the next **`advanceToNextTurn`** after effects like **`grant_command_points_next_turn`**).
- **`GameState.wantedLevelTierIndex`**: index into **`catalog.wantedLevels`** (starts **0**). After **`executePlan`**, **`nextMonotonicWantedTierIndex`** sets **`max(previous index, tier implied by final `player.infamy`)`** so escalation never rolls back when infamy falls. If the new index is **greater** than the previous, opposing agents may spawn up to the new tier’s **`maxAgents`** (see **`spawnOpposingAgentsAfterWantedEscalation`**).
- **`GameState.organizationName`**: display string for the player’s evil organization, chosen once at **`createInitialGameState`** from **`catalog.organizationNames`**.
- **`GameState.activeOmegaPlanId`**: catalog **`OmegaPlanTemplate.id`** for this run, or **`null`** if **`catalog.omegaPlans`** is empty. Chosen once at **`createInitialGameState`** via **`pickRandomOmegaPlanId`**. Shown in the **Omega Plan** panel (three phases, three missions each, names from mission templates). Defines the active **`mapId`** for location scope. **`GameState.activeOmegaStageIndex`** and **`omegaRowProgress`** track which omega **row** must be cleared next; **`executePlan`** sets slot flags when an omega mission in the current stage succeeds and may advance the stage when all three slots succeed.
- **`GameState.currentEventTemplateId`**: **`EventTemplate.id`** for the **current rotating offer**, or **`null`** when **`catalog.events`** is empty. Initialized and re-rolled in **`executePlan`** via **`pickRandomEventTemplateId`**. **`GameState.eventOfferEngagedThisTurn`**: set **`true`** when an **event** mission is assigned in Main; cleared at **`executePlan`** / **`advanceToNextTurn`**. Drives whether **`expireEffects`** fire.
- **`GameState.activeSuccessModifiers`**: `{ delta: number; turnsRemaining: number }[]` — flat **success %** addends from **`add_success_chance_modifier`**; total summed into resolve + UI as **`eventSuccessModifierDelta`**. Each **`executePlan`** decrements **`turnsRemaining`** and drops finished entries **after** optional event expire effects apply.
- **`GameState.activeLairId`** / **`lairMissionIds`** / **`completedLairUpgradeMissionIds`**: random lair at run start when **`catalog.lairs`** is non-empty; **`lairMissionIds`** is the pool for the Lair **Missions** tab (starts as **`availableMissionIds`**). **`completedLairUpgradeMissionIds`** lists upgrade mission template ids that finished **successfully** this run. Assigning from the lair allows **`missionTemplateId`** in **`lairMissionIds`** **or** in **`pendingLairUpgradeMissionIds`** ( **`upgradeMissionIds`** minus **`completedLairUpgradeMissionIds`**).
- **`GameState.locationSecurityStates`**: one **`LocationSecurityState`** per **playable** location this run, initialized at security **`0`**. Shown in the **Locations** panel; **increments on mission resolution** when the resolved mission’s target maps to that location, capped by **`locationLevel`** (see **Locations**).
- **`GameState.locationRequiredTraits`** / **`locationSecurityTraits`**: `Record<locationId, string[]>` — rolled in **`createInitialGameState`** for playable locations only; merged into **`missionSuccessOptionsForTarget`** for mission success and UI when the target has a location id (**`revealedSecurityTraitIds`** applies the security stack slice).
- **`GameState.locationAssetSlots`**: **`LocationAssetPlacement[]`** (each **`locationId`** + **`slots`** of **`LocationAssetSlot`**). Initialized in **`createInitialGameState`** only for playable locations: each gets **1–3** distinct random **`catalog.assets`** ids (none if the asset catalog is empty); all slots start **occupied** and **hidden**, then **`min(3, total slots)`** slots are set **revealed**. The Locations panel shows label **Asset** with value **"Asset"** when hidden, catalog **name** when **occupied** and **revealed**, or **"—"** when **`empty`**; in **Main Phase**, **occupied** rows are **draggable chips** for targeting (`main.ts`).
- **`GameState.opposingAgentInstances`** / **`locationAgentPresence`**: **`AgentInstance[]`** for all spawned opposing agents this run, plus one **`LocationAgentPresence`** row per playable location (**`agentInstanceIds`** empty at start). Each **`AgentInstance`** includes **`catalogVisibility`** (`hidden` \| `revealed`; spawns default **hidden**). **`spawnOpposingAgentsAfterWantedEscalation`** (on wanted tier index increase) appends instances and presence entries; at most one opposing instance per **agent template** id at a time. Designers do not author placement in JSON. **`executePlan`** returns updated **`opposingAgentInstances`** every resolve (reveals at mission sites + tier spawns merge into the same locals).
- **Critical failure (injury)**: If a mission **fails** with **`getMissionTargetLocationId(am.target)`** set and **≥1** opposing agent at that location (count **`all`**, same as the success-roll penalty), **`onFailureEffects`** apply first via **`applyMissionEffects`**, then each **participant** rolls once for the **`injured`** trait with independent chance **`min(100, 20 × agentCount)`**% (**`applyCriticalFailureInjuryRolls`** in **`missionEffects.ts`**; no duplicate trait ids). Targets with **no** mission location id (**`none`** / **`minion`**) never trigger this. **`mission_completed`** logs **`criticalFailure`**, agent count, injury **%**, and which participants gained **`injured`** from that roll pass for the activity panel.
- **`GameState.availableMinionTemplateIds`**: template ids currently offered for hire. Initialized randomly at game start (excluding templates **already on the roster**) and **rerolled at the end of each `executePlan`** with the same exclusion plus **`pickRandomMinionTemplateIds`**. During **Main Phase**, the player may spend **1 CP** (**`rerollHireOffers`**) to redraw the pool immediately. Hiring removes that template id from the list for the rest of the phase until the next reroll. **`hireMinion`** only accepts ids in this list.
- **`ActiveMission`**: `id`, `missionTemplateId`, **`target`** (**`MissionTarget`**), **`missionSource`** (`lair` | `omega` | `event`), **`omegaStageIndex`** / **`omegaSlotIndex`** (set when source is omega; **`null`** for lair and event), `participantInstanceIds` (1–**`maxParticipantsPerMission`** for lair/omega; **1–3** for event), **`plannedAssetIds`** (aligned with template **`requiredAssetIds`** — `null` = empty slot at assign; non-null entries were deducted from **`player.assets`** at assign), `turnsRemaining` (starts at `MissionTemplate.durationTurns`), **`startedOnTurn`**. Missions tick **only** when the player executes the plan: each active mission decrements `turnsRemaining`; at **0**, success is rolled vs **`successChancePercent`** with **`missionSuccessOptionsForTarget(state, target)`**, **`assignedAssetIds: am.plannedAssetIds`** when the template lists required assets (otherwise **`playerAssets: player.assets`** for asset-only missions without slots), **`dynamicTraitDelta`** from **`dynamicTraitSuccessModifierFromFullRoster`**, **`eventSuccessModifierDelta`** from **`activeSuccessModifiers`**, **`opposingAgentPenaltyCount`** from **all** agents at **`getMissionTargetLocationId(target)`** when set, plus **`traitsCatalog: catalog.traits`** (`mission.ts` + **`executePlan`** in **`gameState.ts`**); baseline infamy (−3 success / +5 failure) is applied, then **`onSuccessEffects`** or **`onFailureEffects`** from the template via **`applyMissionEffects`** (**`missionEffects.ts`**), then **`rollDynamicTraitsAfterMission`** updates **`dynamicTraits`** and supplies optional **`mission_completed.dynamicTraitChanges`**. On **success**, **`executePlan`** also applies **`unlock_lair_mission`** entries (append to **`lairMissionIds`**) and, if the resolved mission id is in the active lair’s **`upgradeMissionIds`**, appends it to **`completedLairUpgradeMissionIds`**. **`mission_completed`** logs **`infamyDelta`** = total infamy change (baseline + effect **`infamy_delta`** entries, clamped) plus **`baselineInfamyDelta`** and **`templateEffectDescriptions`** for the activity feed. The same tick applies **participant XP** (and possible **level-ups** + **`minion_leveled_up`** events), **+1 security** at the resolved target location when **`getMissionTargetLocationId(am.target)`** is set (per-location cap), **reveal all opposing agents at that location** when the target maps to a site and at least one agent is present, **`asset_gained`** when a steal effect fires, and **`mission_completed`** logging.
- **`GameState.activityLog`**: **`TurnActivityEntry[]`** (each **`turnNumber`** + **`events`**). Main-phase actions append via **`appendActivityEvent`**; **`executePlan`** batches resolve outcomes with **`mergeResolveActivityEventsIntoActivityLog`**: **`mission_completed`** rows (with **`target: MissionTarget`**, optional **critical failure** injury fields, optional **`dynamicTraitChanges`**) plus any **`minion_leveled_up`** rows (and other **`ActivityEvent`** kinds as features grow). New turn buckets are **prepended** so the **Activity** panel shows **newest turn first**; the log is **not** cleared when the turn advances (session-only persistence until save/load exists). **`ActivityEvent`** union includes at least: `mission_completed`, `mission_started`, `mission_cancelled` (each carries **`target`**), **`mission_aborted`** (`activeMissionId`, `missionTemplateId`, `target`, `reason: "missing_template" | "invalid_participants"` — a resolving mission whose template or roster is invalid is **not silently dropped**: committed `plannedAssetIds` are refunded with `asset_gained` rows and this event is logged; should be unreachable today, exists so future systems fail loudly), `minion_hired`, `minion_rehired`, `minion_fired`, `minion_leveled_up` (`instanceId`, `templateId`, `newLevel`, optional `traitId`), `asset_gained`, `asset_lost`, **`event_rotated_in`**, **`event_expired`**.

Assignment rules: enforced in **`assignMission`** — **`missionTemplate.targetType`** must match the provided **`MissionTarget`**; **playable** location scope for **location** / **asset** targets (**`activeLocationIds`**); **`missionSource`** **`lair`**: mission id must be in **`GameState.lairMissionIds`** **or** in **`pendingLairUpgradeMissionIds`** for the active lair; omega indices null; **`omega`**: active plan, **`activeOmegaStageIndex`**, and mission id must match **`omegaSlotMissionId`** for the chosen slot; **`event`**: **`missionTemplateId`** must equal **`currentEventTemplateId`**, omega indices null, **1–3** participants; sufficient **`startCommandPoints`**; participants not already on a mission; target minion not in participant list; **`activeMissions.length < maxConcurrentMissions`**; **`plannedAssetIds`** must match **`requiredAssetIds`** length, each non-null entry must equal that slot’s required id, and per-id tallies must not exceed **`player.assets`** (deducted on assign, logged as **`asset_lost`**; **`cancelMission`** refunds with **`asset_gained`**).

## Content editor (dev-only)

Open **`http://localhost:5173/editor.html`** while `npm run dev` is running. Vanilla-TS app under **`src/editor/`**; it is **never bundled into production builds** (Vite's build input is `index.html` only, and the API plugin is `apply: "serve"`).

- **Dev API** (plugin in `vite.config.ts`): `GET /__content-api/slices` reads every `CONTENT_MANIFEST` file; `PUT /__content-api/slices` re-runs **`parseContentCatalog`** on the whole draft and **refuses to write when issues exist** (422 with the `ContentIssue[]`), so files on disk always pass the build gate. Writes go through **`serializeContentSlice`** and only touch changed slices. **The first save canonicalizes formatting of every content file once**; after that, diffs are minimal. Saving triggers HMR on the game page (live preview loop). **Art uploads**: `GET /__content-api/art` lists `public/assets/cards/custom/*.webp` (name, path, animated flag); `POST /__content-api/art?name=<slug>[&overwrite=1]` takes raw WebP bytes — filename is slugged, magic bytes verified (a renamed PNG is rejected), ≤ 8 MB, existing file → 409 unless overwriting. Shared pure helpers (slug + `RIFF…WEBP` + `ANIM`-chunk detection) live in **`src/editor/artFiles.ts`** (browser-safe; used by both the plugin and the editor page; unit-tested). Uploads live under `public/`, so Vite serves them in dev, copies them into `dist/` on build, and they are committed to git beside the content that references them.
- **Draft model** (`src/editor/store.ts`): in-memory copy of all raw slices; every edit snapshots (undo, depth 50), re-validates with the real schema pipeline, and mirrors to `localStorage` (restore prompt after an accidental tab close). Invalid work-in-progress lives in the draft, never on disk.
- **Layout** (`src/editor/main.ts`): slice sidebar (issue-count badges) → entity list (`+ New`, per-entity issue dots) → detail form → global issues panel (click an issue to jump to the entity). Header: **Save** (disabled while issues exist), **Undo**, **Unreferenced…** report, status.
- **Forms** (`src/editor/forms/`): hand-written per slice. Every cross-reference field is an **id picker** over the current draft. The **Game Balance** view (`balanceForm.ts`) is a singleton form (no entity list / New / Duplicate / Delete / Rename): grouped fieldsets for every `BalanceConfig` knob, each with a **plain-language tooltip + hint** explaining its gameplay effect, plus a live **formula strip** showing the success-chance formula with the draft's current numbers. Missions/events share one form with the **effects list editor** (kind-specific fields per `MissionEffect`; kinds incompatible with the current `targetType` are disabled via **`effectKindTargetTypeRequirement`**; a note shows the `orderedMissionEffects` resolution order) and a **success-chance preview** (pick up to 3 minion templates; real `computeSuccessChanceBreakdown`). Omega plans render the literal 3×3 mission grid. Minions/agents include a `startingDynamicTraits` sub-editor. Inputs commit on `change` (blur/enter), not per keystroke. Every art-path field (`cardArt` on minions, agents, missions, events, locations, assets, lairs, omega plans; `profilePic` on player profiles) uses the **art field widget** (`src/editor/artField.ts`): live thumbnail preview with an **animated ✓ / static webp / file not found** badge, editable path input, **Upload .webp…** (client-side preflight, name prompt seeded `<slice>-<id>`, overwrite confirm on 409), and a picker over previously uploaded files. **Traits, wanted levels, maps, and organization names intentionally have no art fields.**
- **Tools**: **Rename id…** (uses `applyIdRenameRaw`, updates all referents), **Duplicate** (`-copy` id suffix), **Delete…** with reference-impact modal (`referencesTo` over the draft), **Unreferenced…** report (`unreferencedIds`; caveat shown that unreferenced ≠ unused — some content is only rolled/offered at runtime).
- Entity **id fields are read-only** in forms — ids change only through Rename so references can never silently dangle. New entities prompt for their id.

## Conventions

- **Adding a catalog slice**: add the key to **`CONTENT_SLICE_KEYS`** (drives **`CONTENT_MANIFEST`**, `RawContentSlices`, and the validator CLI), an entity schema + entry in **`contentSliceSchemas`**, a normalizer in **`parseContentSlices`**, semantic checks in **`collectContentIssues`**, the field on **`ContentCatalog`**, the static import in `loadContent.ts`, and (if it holds id references) the walkers in `contentRefs.ts`.
- Report content problems as **`ContentIssue`** rows (accumulate; never throw on the first problem inside the schema layer). Only `parseCatalog` throws, and it throws everything at once.
- Any tool that **writes** `content/*.json` must serialize through **`serializeContentSlice`** so diffs stay canonical.
- **Card art paths** (`cardArt` / `profilePic`) are site-root paths to files under `public/` (uploads go to `public/assets/cards/custom/`, WebP only — animated WebP plays natively in the plain `<img>` from `createCardArtImg`, no code needed). `scripts/validate-content.ts` fails the build when a referenced art file does not exist on disk (Node-only check; the browser schema validates the string shape only).
- Keep **designer data** in `content/*.json` and **runtime/progression** in **`gameState.ts`** and related TS modules (`types.ts`, `minion.ts`, `agent.ts`, `wantedLevel.ts`, `mission.ts`).
- After editing JSON, run **`npm run content:validate`** (or `npm run build`) before assuming content is valid. After editing rules-engine code, run **`npm test`** and **`npm run typecheck`** (the build runs both validation and typecheck; Vite alone does not typecheck).
- Rules-engine functions take an injected **`Rng`** (and an id generator where they mint instances) — never call `Math.random` / `crypto.randomUUID` directly inside `src/game/`.
- **New gameplay constants go in `BalanceConfig`**, not as code literals: add the field + default to `DEFAULT_BALANCE` (`types.ts`), the schema field in `balanceConfigSchema`, thread it from `catalog.balance`, and add the row (label + plain-language tooltip) to the editor's `balanceForm.ts`.
- UI event handlers in `main.ts` route state transitions through **`dispatch`** rather than assigning `state` themselves.
